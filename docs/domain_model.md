# Domain Logic & Schema Overview: Campus

> **Context:** This document explains the business rules, relationships, and data flow built into the `campus-project` database layer (`schema.sql`).

## 1. The Core Hierarchy

The foundation of Campus is its academic structure.
`Universities` -> `Departments` -> `Profiles` (Students)

*   **Universities (`universities`)**: Top-level entity. Name is unique.
*   **Departments (`departments`)**: Child to a university. A department name is only unique *within* a specific university.
*   **Profiles (`profiles`)**: Linked specifically to a `university_id` and `department_id`. Also tracks the student's academic `level`.

## 2. Core Social Features & Business Logic

### 2.1. Posts & Feeds
*   **Table:** `posts`
*   **Logic:** Posts are tied to *both* the `user_id` and the `university_id`. This allows the application to cleanly query "University Feeds" without needing complex cross-joins through the profile table.

### 2.2. Engagement (Likes & Comments)
*   **Likes:** Tracked in the `likes` pivot table.
*   **Trigger Magic:** An SQL trigger (`on_post_like`) on the `likes` table automatically increments or decrements the `likes_count` integer on the `posts` table.
*   **Rule:** **Never manually update `likes_count` from the frontend or a server action.** Simply let the user insert or delete a row in the `likes` table, and the database handles the aggregate count.

### 2.3. Stories (Ephemeral Content)
*   **Table:** `stories`
*   **Logic:** Stories have an `expires_at` timestamp safely defaulting to exactly 24 hours after `created_at`.
*   **Constraint:** A story must have *either* an `image_url` or text `content`. This is enforced at the DB level via `CHECK (image_url IS NOT NULL OR content IS NOT NULL)`.
*   **Styling:** Supports custom text backgrounds natively via the `background_color` column.

### 2.4. Messaging & Groups
*   **1-to-1 Chat:** Managed via the `messages` table with an `is_read` boolean flag.
*   **Groups:** Users can form study/social groups tied to their university. Managed by `groups` and the `group_members` pivot table.

## 3. Security (Row Level Security - RLS)

Supabase RLS is aggressively enabled on all core tables.
*   **Read Access:** Generally public. (e.g., `Profiles`, `Posts`, `Stories`, and `Likes` are viewable by anyone in the app).
*   **Write/Delete Access:** Strictly restricted to the owner (`auth.uid() = user_id`).
*   **Storage Access:** Avatars, Posts, and Stories buckets allow public reads, but only Authenticated users can upload to them.
